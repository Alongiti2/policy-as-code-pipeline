# Policy-as-Code / IaC Security Pipeline

A CI/CD pipeline that blocks non-compliant infrastructure before it ever deploys — Terraform, Checkov, and GitHub Actions, with real (not simulated) scan output proving it works.

## Why this exists

Most of my other repos demonstrate scan-after-the-fact security tooling — finding problems in infrastructure that already exists. This one demonstrates the other half of DevSecOps maturity: **shift-left**, catching the same class of problems in a pull request, before deployment, with automatic enforcement rather than a manual review step. This is also one of the most common Staff-level interview topics in security engineering, so this repo doubles as interview prep, not just a portfolio piece.

## What's actually in this repo

| Folder | What it contains |
|---|---|
| [`terraform/intentionally-misconfigured/`](./terraform/intentionally-misconfigured) | Four resources with real, common misconfigurations (public S3 bucket, open security group, wildcard IAM policy, unencrypted EBS volume) — a test fixture proving the pipeline actually catches things |
| [`terraform/compliant/`](./terraform/compliant) | The same four resources, fixed, with inline comments explaining each fix |
| [`policies/`](./policies) | The Checkov policy baseline used by CI, with every exclusion documented, plus a mapping to CIS AWS Benchmark and NIST 800-53 controls |
| [`.github/workflows/policy-check.yml`](./.github/workflows/policy-check.yml) | The actual GitHub Actions pipeline — blocks merge on policy failure |
| [`examples/`](./examples) | Real (not fabricated) scan output from both the blocked and approved scenarios |

## The evidence is real, not simulated

The scan output in `examples/blocked-pr-scan-output.txt` and `examples/approved-pr-scan-output.txt` was generated by actually running Checkov against the Terraform in this repo — 19 real findings on the misconfigured fixture, 0 findings on the compliant version, using the exact same policy config the CI pipeline uses. You can reproduce both results yourself — see `policies/README.md` for the two commands.

## Why Checkov instead of hand-written OPA/Rego policies

Checkov's built-in policy set already encodes hundreds of maintained checks mapped to standard frameworks (see `policies/README.md`). Reinventing that from scratch as custom Rego would demonstrate less real-world judgment than knowing when a mature, widely-adopted tool is the right call versus building bespoke tooling. The policy *baseline* (which checks are enforced, which are explicitly scoped out and why) is the actual engineering artifact here — that's the part that requires judgment, not the underlying check engine.
